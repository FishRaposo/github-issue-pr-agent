# Key Technical Decisions - GitHub Issue-to-PR Agent

This document details the key technical design choices and trade-offs made in the creation of the GitHub Issue-to-PR Agent.

---

## 1. Directory Sandbox Path Verification

- **Decision**: Restrict file modifications to allowlisted directories (e.g. `src/` and specific development project folders) and explicitly block writes to configuration root paths (like `Makefile` or `pyproject.toml`).
- **Rationale**: An AI agent generating file writes can easily overwrite core files due to hallucinations or prompt injection. Enforcing folder allowlists at the code interface layer prevents catastrophic system corruption.
- **Trade-offs**: Reduces the agent's flexibility (e.g., it cannot dynamically update dependencies in `requirements.txt`). This limit is an acceptable compromise for security.

---

## 2. Draft-Only Pull Requests

- **Decision**: The agent's GitHub client is restricted to creating Draft Pull Requests (`draft=True`) and has **no code merge permissions** or direct branch push access to `main`/`master`.
- **Rationale**: Enforces a strict human-in-the-loop review gate. Code generated by AI must be inspected by a developer before entering the primary repository branch.
- **Trade-offs**: The agent cannot automate the entire deployment pipeline.

---

## 3. Local Subprocess Test Runner (`test_runner.py`)

- **Decision**: Execute test suites using a simple local shell subprocess execution of `pytest` rather than launching dynamic docker test containers.
- **Rationale**: Reduces startup latency and resource overhead during development simulations.
- **Trade-offs**: If tests execute arbitrary or unsafe code, running on the host system exposes the development server to security risks. In production, the test runner must execute inside an ephemeral, isolated Docker container sandbox.

---

## 4. String Replacement Editing Engine (`editor.py`)

- **Decision**: Implement code modifications using precise string search-and-replace instead of generative file overwrites or complex Abstract Syntax Tree (AST) tree-shaking algorithms.
- **Rationale**: Generative AI models often hallucinate line placements when recreating entire code files. Targeting specific block patterns (e.g., replacing standard return statements with guarded returns) ensures highly reliable, predictable patches.
- **Trade-offs**: The agent cannot easily restructure code or implement multi-file refactoring patterns that involve sweeping signature changes.
